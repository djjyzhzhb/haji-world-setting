# 异世界 TRPG 式开放世界设定集 —— 项目构建计划

## 摘要

为用户的架设异世界小说项目建立完整的设定文档目录结构。项目不是单一线性故事，而是作为文学创作基础的 **可运行世界观设定集**。本项目是更大的**人造语言创制计划**的子项目——**目的是为父项目中的人造语言提供文化背景与文明土壤**，使语言获得灵魂与深度。语言本身不在此项目中重新定义，而是通过文化、历史、社会等维度为其赋予血肉。核心美学原则为**广义现实主义**（古希腊 mimesis 观念），强调世界内在逻辑自洽、因果律严格。"合情合理的不可能"优于"不合情合理的可能"。当前处于设定创作阶段。

本计划将创建一个完整的多层目录结构，包含：Git 版本控制、可浏览的 TypeScript 网页界面（`web/`，Vite + TS）、素材投递箱（`_inbox/`）、核心原则文件（AI 运作约束）、模板库、世界观设定（宇宙论/地理/历史/种族/**文化（语言文化的土壤）**/政治/经济/力量体系）、角色卡、叙事模组工具、以及专门的因果律一致性保障体系。不需要严格的数值系统，数值可后填。

## 设计哲学

### 三条核心线索融合

**TRPG 设定集线索**：参考 D&D 三宝书与 Campaign Setting 标准章节编排 —— 从世界观总览到地理、历史、种族、势力、力量体系、生物图鉴、模组的自顶向下结构。世界是可被探索的，而非仅供被动阅读。

**广义现实主义（Mimesis）线索**：基于亚里士多德《诗学》—— 摹仿不是复制现实，而是揭示事物的内在本质和因果必然性。世界中的每一个事件都必须有前因，每一种社会结构必须有物质/历史条件支撑，所有超自然力量必须有边界和代价。

**AI 辅助运作线索**：所有文档为纯文本 Markdown 格式，配备标准 YAML front matter；设立专门的 `_ai_context` 和 `_consistency_rules` 等 AI 关键文件；配备完整的模板库确保格式一致。

## 关键设计决策

### 为什么角色目录（09）排在地理/历史/种族/文化/政治/经济之后？

传统小说项目常将角色前置。但本项目本质是 TRPG 设定集式的**可运行世界**——世界先于人物存在，人物是世界条件的产物。这与广义现实主义一致：角色由其社会环境、经济条件、历史背景塑造，而非反过来。

### 为什么需要独立的 `13_因果律与一致性/` 目录？

在常规世界构建中，一致性规则散布各处。但以 mimesis 为核心约束的项目，一致性不是附加品而是**骨架**。集中管理使 AI 在每次生成后可以系统性对照检查。

### 为什么 `05_文化与社会/` 是语言的文化桥梁而非语言创制模块？

本项目是**人造语言创制计划**的子项目，但语言本身（音系、语法、词汇）已在父项目中完整定义。本项目的角色是**为这门语言提供文化土壤**——构建使用该语言的文明的历史、社会、宗教、文学传统，使语言不再是空洞的规则集合，而是有血有肉的文化载体。`05_文化与社会/01_语言文化桥梁.md` 作为父语言项目的文化接口，记录语言在文明中的角色：谁在使用它、在什么场合使用、它如何塑造世界观、有哪些文学传统和谚语——这些是赋予语言灵魂的内容，而非语言规则本身。

### 为什么 `_ai_context.md` 放在根目录而非 `00_项目总览/`？

AI 需要最高便捷性的"快速启动文件"。根目录位置确保最高可见性。内容控制在 500 字以内，足以让 AI 在数秒内理解项目全貌。

### 为什么需要 TypeScript 网页项目？

大量 Markdown 文件在文件管理器中浏览体验差。一个基于 Vite + TypeScript 的单页应用提供侧边栏导航、搜索和 Markdown 渲染，既可在本地 `npm run dev` 热更新预览，也可 `npm run build` 构建后部署到 GitHub Pages 或本地双击打开。GitHub 仓库命名为 `haji-world-setting`，遵循用户现有命名约定（`haji-vocabulary-estimator`、`haji-astronomy-model`）。

### 为什么 `_inbox/` 是用户的素材入口而非自动整理？

`_inbox/` 是用户和 AI 之间的"交接区"。用户可能随时产生灵感（一张截图、一段语音转文字、一个链接），需要一个低摩擦的入口随手丢入。AI 的职责是定期检视并将其整理到正确位置——这保持了项目结构的整洁，同时不限制用户的输入方式。

### 为什么 Git 放在阶段零（最优先）？

Git 初始化必须在任何文件创建之前完成，否则后续的 `.gitignore` 配置和 `.gitkeep` 文件会混杂在首次提交中。优先初始化确保从第一个文件起就有完整的版本历史。

### 文件格式约定

- 全部使用 **Markdown (.md)**
- 每个设定文档含 **YAML front matter** 元数据头
- 支持 `[[双向链接]]` 语法（Obsidian 兼容）
- 文件名前缀序号保证文件系统排序

### YAML Front Matter 标准

```yaml
---
id: DOC-XXX              # 文档唯一ID
title: ""                # 标题
category: ""             # 所属分类
tags: []                 # 标签
created: YYYY-MM-DD      # 创建日期
modified: YYYY-MM-DD     # 最后修改日期
status: draft | review | final
causal_refs: []          # 关联的因果文档ID
---
```
