# 远程协作与变更文档系统 —— 计划（简化版）

## 摘要

三项能力：(1) 推送到 GitHub + GitHub Pages 部署网页；(2) 前端浏览器内生成变更文档 → 下载 JSON 文件；(3) 本地 AI 读取 `_changelog/` 目录理解用户意图。无需后端服务器，无需数据库。

---

## 当前状态分析

- Git 已本地初始化，未关联远程仓库（repo 名 `haji-world-setting`）
- 网页是纯前端 Vite 项目，`npm run build` 输出到 `web/dist/`
- 前端目前只能读取和展示 Markdown 文件，不能生成变更文档
- 无数据库，不需要数据库

---

## 核心设计理念

### 纯前端 + 下载模式

```
浏览器前端操作 → 生成结构化 JSON 变更文档 → 浏览器下载 JSON 文件
                                              ↓
                        用户手动放入 _changelog/pending/  ← 本地电脑
                                              ↓
                        AI 读取 _changelog/  ← 理解用户意图并处理
```

无需后端，无需数据库。GitHub Pages 托管静态网页，前端生成变更文档，用户下载后放入本地对应目录。

## 实施计划

### 阶段一：Git 推送 + GitHub Pages

| 序号 | 操作 | 说明 |
|------|------|------|
| 1a | 在 GitHub 创建仓库 `djjyzhzhb/haji-world-setting` | 用户手动创建 |
| 1b | `git remote add origin` | 关联远程仓库 |
| 1c | `git add -A && git commit -m "项目骨架初始化"` | 首次提交 |
| 1d | `git push -u origin main` | 推送到 GitHub |
| 1e | GitHub Pages 设置 | 仓库 Settings → Pages → Source: GitHub Actions / `main` 分支 `web/dist/` |
| 1f | 配置 `.github/workflows/deploy.yml` | 自动构建并部署到 GitHub Pages |

### 阶段二：前端变更标注系统（`web/src/`）

| 序号 | 文件 | 内容 |
|------|------|------|
| 2a | `web/src/api.ts` | 新建：变更文档数据模型 TypeScript 类型定义 |
| 2b | `web/src/ui/changes.ts` | 新建：变更面板 UI 组件 |
| 2c | `web/src/main.ts` | 修改：集成标注按钮、变更面板、下载逻辑 |
| 2d | `web/src/style.css` | 修改：新增标注按钮、面板、下载按钮样式 |

**前端新增功能**：
- 每个段落旁显示 `+` 标注按钮（hover 时显现）
- 点击后弹出标注面板：选择类型（标注/草稿/修正/灵感）、填写标题和内容
- 底部"变更记录"面板：查看当前会话中已生成的所有变更
- 每条变更旁有"下载"按钮 → 下载为 `CHG-YYYYMMDD-NNN.json` 文件
- 支持"全部下载"→ 打包为 ZIP 或逐条下载

### 阶段三：变更文档格式（`_changelog/`）

在项目根目录新建 `_changelog/` 目录：

```
_changelog/
├── README.md                   # 使用说明
├── pending/                    # 待处理的变更
│   └── .gitkeep
├── reviewed/                   # 已审阅
│   └── .gitkeep
├── applied/                    # 已应用
│   └── .gitkeep
└── rejected/                   # 已拒绝
    └── .gitkeep
```

变更文档 JSON 格式：

```json
{
  "id": "CHG-20260612-001",
  "type": "annotation",
  "author": "user",
  "timestamp": "2026-06-12T14:30:00+08:00",
  "target": {
    "docPath": "05_文化与社会/01_语言文化桥梁.md",
    "section": "## 语言与世界观"
  },
  "content": {
    "summary": "关于哈吉语对颜色认知的影响",
    "body": "哈吉语中颜色词汇的分化反映了...",
    "tags": ["颜色", "认知", "语言"]
  },
  "status": "pending"
}
```

### 阶段五：用户工作流

```
1. 手机/任意设备打开 https://djjyzhzhb.github.io/haji-world-setting/
2. 浏览设定文档，点击段落旁的 + 按钮
3. 填写标注（灵感/修正/草稿/标注）
4. 点击下载 → JSON 文件保存到手机
5. 回到电脑后，将 JSON 文件放入 _changelog/pending/
6. AI 下次会话时读取 _changelog/pending/ → 处理变更
```

## 关键设计决策

### 为什么下载而不是在线写入？

GitHub Pages 是纯静态托管，前端无法直接写入文件系统。下载 JSON 文件是最简单的方案——用户只需将文件放入本地 `_changelog/pending/` 目录即可。零后端依赖，零数据库。

### 为什么不需要后端？

- 单用户场景，不需要实时同步
- 变更文档数量少，下载+手动放入的成本极低
- 去除了 Express 依赖、端口管理、局域网配置等所有运维负担
- GitHub Pages 免费托管，全球可访问

### 为什么需要 GitHub Actions 自动部署？

手动 `npm run build` + push 每次都要两步。GitHub Actions 可以做到 push 即自动构建部署，`.github/workflows/deploy.yml` 配置一次即可。
